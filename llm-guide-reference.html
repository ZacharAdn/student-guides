קונספט 1. הנדסת פרומפט — Prompt Engineering

System Prompt vs User Prompt

ה-System Prompt מגדיר את ה"אישיות" וההתנהגות של המודל — הוא נטען פעם אחת בתחילת השיחה ומשפיע על כל התשובות. ה-User Prompt הוא ההודעה הספציפית שהמשתמש שולח.

python 
import google.generativeai as genai

genai.configure(api_key="YOUR_API_KEY")

model = genai.GenerativeModel(
    model_name="gemini-2.0-flash",
    system_instruction="""You are an expert agronomist AI assistant
for SolarWine, specializing in agrivoltaics (solar panels + agriculture).
Always respond in Hebrew. When discussing technical measurements,
include both metric and imperial units.
Keep responses concise and actionable."""
)

chat = model.start_chat()
response = chat.send_message("מה ההשפעה של פאנלים סולאריים על גידול גפנים?")
print(response.text)

Few-Shot Learning

הוספת דוגמאות ל-Prompt כדי "ללמד" את המודל את הפורמט הרצוי. ככלל, 3 דוגמאות מספיקות:

python 
prompt = """Classify the following sensor readings as Normal or Alert.

Examples:
- Soil moisture: 45%, Temperature: 28°C, Light: 650 lux → Normal
- Soil moisture: 12%, Temperature: 41°C, Light: 1200 lux → Alert
- Soil moisture: 38%, Temperature: 22°C, Light: 400 lux → Normal

Now classify:
- Soil moisture: 15%, Temperature: 39°C, Light: 1100 lux →"""

response = model.generate_content(prompt)
print(response.text)  # → Alert

בקשה מהמודל לחשוב שלב אחר שלב לפני שהוא נותן תשובה סופית. משפר דרמטית את הדיוק בבעיות מורכבות:

Best Practices — צ'קליסט

  • הגדר תפקיד ברור ב-System Prompt (מי אתה, מה אתה יודע)
  • היה ספציפי — "ענה ב-3 נקודות" עדיף על "ענה בקצרה"
  • השתמש ב-Few-Shot כשאתה צריך פורמט ספציפי
  • הוסף "Think step by step" לבעיות מורכבות
  • Temperature 0 לניתוח נתונים, 0.7-1.0 ליצירתיות
  • בדוק את ה-Prompt על מקרי קצה לפני שמכניסים לפרודקשן
  • שמור Prompt templates בקובץ נפרד, לא hardcoded
  • נסח בשלילה כשצריך: "אל תמציא נתונים" לצד "ענה רק על סמך המידע"

קונספט 3. כוונון עדין — Fine-Tuning

מתי כן ומתי לא?

כוונון עדין הוא לא תמיד הפתרון. לפני שמתחילים, צריך לעבור את עץ ההחלטות:

כלל אצבע

התחל תמיד עם Prompt Engineering. אם לא מספיק — נסה RAG. רק אם שניהם לא עובדים — שקול Fine-Tuning. זה חוסך זמן, כסף, ותחזוקה.

LoRA — הסבר פשוט

LoRA (Low-Rank Adaptation) מאפשר לכוונן מודל ענק בלי לשנות את כל המשקולות שלו.

אנלוגיה: דמיין שיש לך אנציקלופדיה של 30 כרכים (= המודל המקורי). במקום לכתוב אנציקלופדיה חדשה, אתה מוסיף דפי Post-it בנקודות ספציפיות (= LoRA adapters). הקורא רואה את הטקסט המקורי + ההערות שלך, בלי שהאנציקלופדיה המקורית השתנתה.

פרמטרים של LoRA

פרמטר מה עושה ברירת מחדל המלצה
r (rank)גודל ה-adapter — כמה "דפי Post-it"88-32 לרוב, 64+ לדאטה מורכב
lora_alphaמקדם scaling — כמה "חזק" ההשפעה של ה-adapter16בד"כ alpha = 2 * rank
lora_dropoutRegularization — מונע overfitting0.00.05-0.1 חובה!
target_modulesאילו שכבות לכוונןvaries"q_proj", "v_proj" מינימלי
טעויות נפוצות (מניסיון!)

Overfitting: אם המודל "שינן" את הדאטה במקום ללמוד את הדפוס — בדרך כלל בגלל: אין dropout, אין scheduler, דאטה לא מגוון מספיק, או train על חודש אחד בלבד (כמו רק מאי).

פתרון: תמיד lora_dropout=0.05+, lr_scheduler_type="cosine", ודאטה מגוון שמייצג את כל התנאים.

קוד — LoRA עם PEFT

python 
from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments
from trl import SFTTrainer

# Load base model
model_name = "meta-llama/Llama-3.2-3B-Instruct"
model = AutoModelForCausalLM.from_pretrained(model_name, torch_dtype="auto")
tokenizer = AutoTokenizer.from_pretrained(model_name)

# Configure LoRA
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                          # rank — balance between capacity and efficiency
    lora_alpha=32,                 # scaling factor (usually 2x rank)
    lora_dropout=0.05,             # CRITICAL: prevents overfitting
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    bias="none"
)

# Apply LoRA
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# → trainable params: 6.5M || all params: 3.2B || trainable%: 0.20%

# Training arguments
training_args = TrainingArguments(
    output_dir="./lora_output",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    learning_rate=2e-4,
    lr_scheduler_type="cosine",    # IMPORTANT: gradual LR decay
    warmup_ratio=0.1,
    logging_steps=10,
    save_strategy="epoch",
    fp16=True
)

# Train with SFTTrainer (simplified fine-tuning)
trainer = SFTTrainer(
    model=model,
    args=training_args,
    train_dataset=train_dataset,   # your formatted dataset
    tokenizer=tokenizer,
    max_seq_length=512
)
trainer.train()

# Then apply LoRA on top — same as before

פריימוורק 4. RAG — שליפה מועשרת

מה זה RAG?

Retrieval-Augmented Generation — במקום ש-LLM ינסה "לזכור" מידע מהאימון, אנחנו מוצאים את המידע הרלוונטי ומזריקים אותו ל-Prompt. המודל מקבל קונטקסט עדכני ומדויק.

Chunking Strategies — אסטרטגיות חלוקה

  • Fixed Size: חלוקה לפי מספר תווים/טוקנים קבוע עם overlap. פשוט אבל עלול לחתוך באמצע משפט.
  • Recursive Character: מנסה לחלק לפי פסקאות, אז משפטים, אז מילים. הכי נפוץ ומומלץ להתחלה.
  • Semantic: חלוקה לפי embedding similarity — כל chunk הוא "נושא" אחד. הכי חכם, הכי יקר.

קוד — RAG Pipeline מלא עם ChromaDB + Gemini

bash

python 
import chromadb
import google.generativeai as genai
import os

genai.configure(api_key=os.getenv("GEMINI_API_KEY"))

# ── Step 1: Create vector store and add documents ──
client = chromadb.Client()
collection = client.create_collection(
    name="agrivoltaics_docs",
    metadata={"hnsw:space": "cosine"}
)

# Your domain knowledge
documents = [
    "Agrivoltaic systems can increase land use efficiency by 60-70% compared to separate solar and agricultural use.",
    "Grapevines under solar panels show 15-20% reduction in water stress due to partial shading.",
    "Optimal panel height for vineyard agrivoltaics is 4-5 meters to allow machinery access.",
    "The PAR (Photosynthetically Active Radiation) under panels is typically 50-70% of full sunlight.",
    "Soil moisture retention under panels increases by 10-30% depending on panel density and climate.",
    "Wine grape quality metrics (Brix, pH, TA) show minimal impact at 30% shading levels.",
    "Bifacial solar panels are preferred for agrivoltaics due to ground-reflected light capture.",
    "Annual energy yield for agrivoltaic systems is 5-10% lower than ground-mounted PV due to higher mounting."
]

# Add to collection (ChromaDB handles embedding automatically)
collection.add(
    documents=documents,
    ids=[f"doc_{i}" for i in range(len(documents))]
)

# ── Step 2: Query and retrieve ──
def rag_query(question: str, n_results: int = 3) -> str:
    """Query the knowledge base and generate an answer."""

    # Retrieve relevant documents
    results = collection.query(
        query_texts=[question],
        n_results=n_results
    )

    context = "\n".join(results["documents"][0])

    # Build RAG prompt
    prompt = f"""Based on the following context, answer the question.
If the context doesn't contain enough information, say so.

Context:
{context}

Question: {question}

Answer in Hebrew, citing specific data from the context:"""

    # Generate answer with Gemini
    model = genai.GenerativeModel("gemini-2.0-flash")
    response = model.generate_content(prompt)
    return response.text

# ── Step 3: Use it ──
answer = rag_query("מה ההשפעה של פאנלים סולאריים על לחות הקרקע?")
print(answer)

קונספט 5. צ'אטבוט עם זיכרון — Chatbot with Memory

למה זיכרון חשוב?

LLM הוא stateless — כל קריאה ל-API היא בלתי תלויה. בלי זיכרון, הצ'אטבוט "שוכח" את כל השיחה אחרי כל הודעה. זיכרון מאפשר המשכיות, הפניה לנושאים קודמים, ושיחה טבעית.

סוגי זיכרון

סוגמה עושהיתרונותחסרונות

קוד — צ'אטבוט עם Gemini + זיכרון

python 
import google.generativeai as genai
import json
import os

genai.configure(api_key=os.getenv("GEMINI_API_KEY"))

class ChatBot:
    """Simple chatbot with conversation memory and session persistence."""

    def __init__(self, system_prompt: str, memory_file: str = "chat_memory.json"):
        self.system_prompt = system_prompt
        self.memory_file = memory_file
        self.history = self._load_history()

        self.model = genai.GenerativeModel(
            "gemini-2.0-flash",
            system_instruction=system_prompt
        )
        self.chat = self.model.start_chat(history=self.history)

    def _load_history(self) -> list:
        """Load conversation history from file."""
        if os.path.exists(self.memory_file):
            with open(self.memory_file, "r") as f:
                data = json.load(f)
            # Convert to Gemini format
            return [
                {"role": msg["role"], "parts": [msg["content"]]}
                for msg in data
            ]
        return []

    def _save_history(self):
        """Save conversation history to file."""
        data = [
            {"role": msg.role, "content": msg.parts[0].text}
            for msg in self.chat.history
        ]
        with open(self.memory_file, "w") as f:
            json.dump(data, f, ensure_ascii=False, indent=2)

    def send(self, message: str) -> str:
        """Send a message and get a response with full context."""
        response = self.chat.send_message(message)
        self._save_history()
        return response.text

    def get_history_length(self) -> int:
        """How many messages in the conversation."""
        return len(self.chat.history)


# ── Usage ──
bot = ChatBot(
    system_prompt="""You are SolarWine's internal assistant.
You help team members with agrivoltaics data, sensor readings,
and vineyard management. Always respond in Hebrew.
When citing data, mention the source or sensor ID."""
)

# Conversation with memory
print(bot.send("מה הטמפרטורה הממוצעת בחלקה 3 השבוע?"))
print(bot.send("ומה עם הלחות שם?"))  # Bot remembers we're talking about plot 3
print(bot.send("תשווה את זה לשבוע שעבר"))  # Bot remembers both metrics

print(f"Messages in memory: {bot.get_history_length()}")

Window Memory — כשהשיחה ארוכה מדי

python 
class WindowChatBot(ChatBot):
    """Chatbot that keeps only the last N message pairs."""

    def __init__(self, system_prompt: str, window_size: int = 10, **kwargs):
        self.window_size = window_size
        super().__init__(system_prompt, **kwargs)

    def send(self, message: str) -> str:
        response = self.chat.send_message(message)

        # Trim history to window size (keep last N pairs)
        if len(self.chat.history) > self.window_size * 2:
            self.chat.history = self.chat.history[-(self.window_size * 2):]

        self._save_history()
        return response.text

# Keeps last 5 exchanges (10 messages)
bot = WindowChatBot("You are a helpful assistant.", window_size=5)

מתי להשתמש בכל סוג?

  • Buffer: שיחות קצרות (עד ~20 הודעות), כשכל פרט חשוב
  • Window: צ'אט תמיכה, שאלות ותשובות — בד"כ 5-10 הודעות אחרונות מספיקות
  • Summary: שיחות ארוכות שצריכות הקשר אבל לא כל מילה
  • Vector: כשיש מאות/אלפי שיחות ורוצים למצוא קשרים בין נושאים

פריימוורק 6. סוכנים ופריימוורקים — Agents

LLM vs Agent — הגבול

LLM = one-shot: נותנים prompt, מקבלים תשובה. זהו.

Agent = LLM + כלים + לולאה: ה-Agent מחליט אילו כלים להפעיל, בודק תוצאות, וממשיך עד שהמשימה הושלמה.

┌──────────┐ │ User │ │ Query │ └────┬─────┘ │ ┌────▼─────┐ ┌────→│ LLM │←────┐ │ │ (Think) │ │ │ └────┬─────┘ │ │ │ │ │ ┌──────▼──────┐ │ │ │ Decide │ │ │ │ next step │ │ │ └──────┬──────┘ │ │ │ │ │ ┌──────▼──────┐ │ │ │ Tool / API │ │ │ │ call │────┘ │ └──────┬──────┘ │ │ │ ┌──────▼──────┐ │ │ Observe │ │ │ result │ │ └──────┬──────┘ │ │ │ Done? ─┤ │ No ───→┘ │ Yes │ │ │ ┌──────▼──────┐ └ │ Final │ │ Answer │ └─────────────┘

Tool Use / Function Calling

Function Calling מאפשר ל-LLM "לקרוא" לפונקציות חיצוניות. המודל לא מריץ קוד — הוא מחזיר JSON שאומר "תפעיל את הפונקציה הזו עם הפרמטרים האלה", ואתה מריץ ומחזיר לו את התוצאה.

python 
import google.generativeai as genai
import os

genai.configure(api_key=os.getenv("GEMINI_API_KEY"))

# Define tools the model can use
def get_sensor_reading(sensor_id: str, metric: str) -> dict:
    """Get the latest reading from a farm sensor.

    Args:
        sensor_id: The sensor identifier (e.g., "PLOT3_SOIL_01")
        metric: The metric to read ("temperature", "moisture", "par")
    """
    # In production, this would query your actual sensor database
    fake_data = {
        "PLOT3_SOIL_01": {"temperature": 24.5, "moisture": 38.2, "par": 580},
        "PLOT3_SOIL_02": {"temperature": 23.8, "moisture": 41.0, "par": 520},
    }
    reading = fake_data.get(sensor_id, {}).get(metric, "Sensor not found")
    return {"sensor_id": sensor_id, "metric": metric, "value": reading, "unit": "varies"}

def get_weather_forecast(location: str, days: int) -> dict:
    """Get weather forecast for a location.

    Args:
        location: Location name (e.g., "Negev Farm")
        days: Number of days to forecast (1-7)
    """
    return {
        "location": location,
        "forecast": [
            {"day": 1, "temp_high": 32, "temp_low": 18, "rain_prob": 0.05},
            {"day": 2, "temp_high": 34, "temp_low": 19, "rain_prob": 0.0},
        ][:days]
    }

# Create model with tools
model = genai.GenerativeModel(
    "gemini-2.0-flash",
    tools=[get_sensor_reading, get_weather_forecast],
    system_instruction="You are SolarWine's farm monitoring assistant. Use tools to answer questions about sensors and weather. Respond in Hebrew."
)

chat = model.start_chat()

# The model will decide which tool(s) to call
response = chat.send_message("מה הטמפרטורה בחלקה 3? וגם מה תחזית מזג האוויר ל-3 ימים?")

# Handle function calls
for part in response.parts:
    if fn := part.function_call:
        print(f"Model wants to call: {fn.name}({dict(fn.args)})")

        # Execute the function
        if fn.name == "get_sensor_reading":
            result = get_sensor_reading(**dict(fn.args))
        elif fn.name == "get_weather_forecast":
            result = get_weather_forecast(**dict(fn.args))

        # Send result back to the model
        response = chat.send_message(
            genai.protos.Content(
                parts=[genai.protos.Part(
                    function_response=genai.protos.FunctionResponse(
                        name=fn.name,
                        response={"result": result}
                    )
                )]
            )
        )

print(response.text)

python 
from langgraph.graph import StateGraph, END
from typing import TypedDict, Literal

# Define the state schema
class SensorAlertState(TypedDict):
    sensor_data: dict
    severity: str      # "low", "medium", "high"
    action: str        # what action to take
    notification: str  # message to send

# Define nodes (each node is a function)
def analyze_reading(state: SensorAlertState) -> SensorAlertState:
    """Analyze sensor data and determine severity."""
    data = state["sensor_data"]
    moisture = data.get("moisture", 50)
    temp = data.get("temperature", 25)

    if moisture < 15 or temp > 40:
        state["severity"] = "high"
    elif moisture < 25 or temp > 35:
        state["severity"] = "medium"
    else:
        state["severity"] = "low"
    return state

def decide_action(state: SensorAlertState) -> SensorAlertState:
    """Decide what action to take based on severity."""
    actions = {
        "high": "IMMEDIATE: Activate emergency irrigation + notify team",
        "medium": "SCHEDULE: Increase irrigation by 20% tomorrow morning",
        "low": "LOG: Normal reading, no action needed"
    }
    state["action"] = actions[state["severity"]]
    return state

def send_notification(state: SensorAlertState) -> SensorAlertState:
    """Prepare notification message."""
    state["notification"] = (
        f"Alert [{state['severity'].upper()}]: "
        f"Sensor {state['sensor_data'].get('sensor_id', 'unknown')} - "
        f"{state['action']}"
    )
    return state

# Define routing logic
def should_notify(state: SensorAlertState) -> Literal["notify", "end"]:
    return "notify" if state["severity"] in ("high", "medium") else "end"

# Build the graph
graph = StateGraph(SensorAlertState)
graph.add_node("analyze", analyze_reading)
graph.add_node("decide", decide_action)
graph.add_node("notify", send_notification)

graph.set_entry_point("analyze")
graph.add_edge("analyze", "decide")
graph.add_conditional_edges("decide", should_notify, {
    "notify": "notify",
    "end": END
})
graph.add_edge("notify", END)

# Compile and run
app = graph.compile()
result = app.invoke({
    "sensor_data": {"sensor_id": "PLOT3_SOIL_01", "moisture": 12, "temperature": 38},
    "severity": "",
    "action": "",
    "notification": ""
})
print(result["notification"])
# → Alert [HIGH]: Sensor PLOT3_SOIL_01 - IMMEDIATE: Activate emergency irrigation + notify team

Rate Limiting

python 
from functools import wraps
import time
from collections import defaultdict

class RateLimiter:
    """Simple in-memory rate limiter."""

    def __init__(self, max_requests: int = 10, window_seconds: int = 60):
        self.max_requests = max_requests
        self.window = window_seconds
        self.requests = defaultdict(list)

    def is_allowed(self, user_id: str) -> bool:
        now = time.time()
        # Remove old requests outside the window
        self.requests[user_id] = [
            t for t in self.requests[user_id]
            if now - t < self.window
        ]

        if len(self.requests[user_id]) >= self.max_requests:
            return False

        self.requests[user_id].append(now)
        return True

limiter = RateLimiter(max_requests=10, window_seconds=60)
