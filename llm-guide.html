🧠 What is an LLM?

A Large Language Model is a neural network trained on massive text data that can understand and generate human language. It predicts the most likely next token (word piece) given a sequence — and from that simple principle, emergent abilities like reasoning, coding, and conversation arise.

🧠

Super-powered Autocomplete

Like your phone keyboard predicting the next word — but trained on billions of documents and thousands of times more powerful.

📚

Read the Entire Internet

Imagine someone who read every book, article, and forum post ever written — and can recall patterns from all of it.

🎭

Role Player

Give it a system prompt and it becomes any character — an agronomist, a code reviewer, a Hebrew teacher. The prompt is the script.

🔮

Pattern Predictor

At its core, it predicts the next token. But from this simple mechanism, complex reasoning and creativity emerge.

Prompt Engineering in Practice

Here are complete, runnable code examples for each technique — using SolarWine agrivoltaics scenarios:

System vs User Prompt

The system prompt defines the model's personality and rules. The user prompt is the specific task. Separating them gives you consistent behavior across all queries:

PYTHON 
import google.generativeai as genai

model = genai.GenerativeModel(
    "gemini-2.0-flash",
    system_instruction="You are an expert agronomist specializing in grapevines under solar panels. Answer only with numbers and data. Be concise."
)

response = model.generate_content(
    "What is the optimal PAR range for Cabernet Sauvignon?"
)
print(response.text)

Few-Shot Learning

Provide 3-5 input/output examples and the model learns the pattern. No training needed — the examples are the instruction:

PYTHON 
prompt = """Classify sensor readings into plant stress levels:

Tleaf=28°C, VPD=1.2kPa, PAR=800 → Low stress
Tleaf=35°C, VPD=2.8kPa, PAR=1200 → High stress
Tleaf=31°C, VPD=1.8kPa, PAR=600 → Medium stress

Tleaf=38°C, VPD=3.5kPa, PAR=1400 →"""

response = model.generate_content(prompt)

Chain of Thought

Force the model to reason step by step. This dramatically improves accuracy on complex analytical tasks:

PYTHON 
prompt = """Analyze step by step why photosynthesis dropped:

Sensor data:
- Yesterday: A=12.5, Tleaf=29°C, PAR=950
- Today: A=6.2, Tleaf=37°C, PAR=1100

Step 1: Compare the values...
Step 2: Identify which changed most...
Step 3: Explain the likely cause..."""

Parameters Checklist

✅ Temperature 0 for data analysis, 0.7-1.0 for creative content generation

✅ Test your prompt on edge cases before deploying to production

✅ Store prompt templates in separate files, not hardcoded in your application code

✅ Use grounding instructions: "don't invent data" + "answer only from the provided context"

Fine-Tuning in Practice — LoRA

Think of fine-tuning like customizing an encyclopedia. The encyclopedia (base model) already knows everything — but you want it to answer in your style, using your terminology. Instead of rewriting the entire encyclopedia (full fine-tuning = expensive), LoRA (Low-Rank Adaptation) attaches small "Post-it notes" to specific pages. These tiny adapters change how the model responds without modifying the original weights. Result: 0.1-1% of parameters trained, 95%+ of the quality.

LoRA Parameters Reference

Parameter What it does Default Recommendation
r (rank) Adapter size — how many "Post-it" pages 8 8-32 for most, +64 for complex data
lora_alpha Scaling factor — how "strong" the adapter influence is 16 Usually 2 * rank
lora_dropout Regularization — prevents overfitting 0.0 0.05-0.1 mandatory!
target_modules Which layers to tune varies "q_proj", "v_proj" minimum

LoRA Fine-Tuning Code with PEFT

PYTHON 
from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments
from trl import SFTTrainer

# Load base model
model_name = "meta-llama/Llama-3.2-3B-Instruct"
model = AutoModelForCausalLM.from_pretrained(model_name, torch_dtype="auto")
tokenizer = AutoTokenizer.from_pretrained(model_name)

# Configure LoRA
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,
    lora_alpha=32,
    lora_dropout=0.05,    # CRITICAL: prevents overfitting
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    bias="none"
)

# Apply LoRA
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# → trainable params: 6.5M || all params: 3.2B || trainable%: 0.20%

# Training arguments
training_args = TrainingArguments(
    output_dir="./lora_output",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    learning_rate=2e-4,
    lr_scheduler_type="cosine",
    warmup_ratio=0.1,
    logging_steps=10,
    save_strategy="epoch",
    fp16=True
)

trainer = SFTTrainer(
    model=model,
    args=training_args,
    train_dataset=train_dataset,
    tokenizer=tokenizer,
    max_seq_length=512
)
trainer.train()
⚠️ Common Fine-Tuning Mistakes

Overfitting is the #1 killer of fine-tuning projects. Signs: model repeats training examples verbatim, fails on new inputs.
Prevention: always use lora_dropout=0.05+, set lr_scheduler_type="cosine", and make sure your training data represents ALL conditions (not just sunny days for SolarWine!).

Full RAG Pipeline — Complete Example

Here is a complete, end-to-end RAG pipeline you can run locally. It loads documents, stores them in ChromaDB, and answers questions using Gemini with retrieved context:

PYTHON 
import chromadb
import google.generativeai as genai
import os

# 1. Setup
genai.configure(api_key=os.getenv("GEMINI_API_KEY"))
client = chromadb.Client()
collection = client.create_collection("solarwine_docs")

# 2. Load and chunk documents
documents = [
    "Agrivoltaics combines solar energy production with agriculture...",
    "The Farquhar model describes photosynthesis as a function of...",
    "Panel tilt angle affects PAR transmission to crops below..."
]

# 3. Add to vector store (ChromaDB auto-embeds with default model)
collection.add(
    documents=documents,
    ids=[f"doc_{i}" for i in range(len(documents))]
)

# 4. Query
results = collection.query(
    query_texts=["How do solar panels affect photosynthesis?"],
    n_results=2
)
context = "\n".join(results["documents"][0])

# 5. Generate with context
model = genai.GenerativeModel("gemini-2.0-flash")
response = model.generate_content(
    f"Based on this context:\n{context}\n\nQuestion: How do solar panels affect photosynthesis?\nAnswer in Hebrew:"
)
print(response.text)
💡 RAG Pro Tips

Start with ChromaDB for prototyping — zero setup, runs in-memory. When moving to production, consider pgvector (if you already use Supabase) or Pinecone (for auto-scaling). Always test retrieval quality before blaming the LLM — bad retrieval = bad answers.

🤖 Agents & Frameworks

When an LLM gains the ability to use tools and make decisions in a loop, it becomes an Agent. This is where AI goes from "answering questions" to "getting things done."

When Does an LLM Become an Agent?

🧠

LLM = Brain

Thinks, reasons, decides what to do next. The intelligence layer.

🔧

Tools = Hands

APIs, databases, sensors, calculators — anything the agent can call to act on the world.

🔄

Loop = Persistence

The agent keeps going: observe → think → act → observe... until the task is done.

🎯

Goal = Mission

Unlike a chatbot that answers once, an agent pursues a goal across multiple steps.

Gemini Function Calling — Your First Agent

Function calling lets the LLM decide which tool to use and what arguments to pass. You define the tools, Gemini orchestrates them:

PYTHON 
import google.generativeai as genai

# Define tools the agent can use
def get_sensor_data(block_id: str, metric: str) -> dict:
    """Get latest sensor reading for a vineyard block."""
    # In production: query your actual sensor API
    return {"block": block_id, "metric": metric, "value": 28.5, "unit": "°C"}

def calculate_stress(tleaf: float, vpd: float) -> dict:
    """Calculate plant stress level from leaf temperature and VPD."""
    if tleaf > 35 and vpd > 2.5:
        return {"level": "high", "recommendation": "Increase shading"}
    elif tleaf > 30:
        return {"level": "medium", "recommendation": "Monitor closely"}
    return {"level": "low", "recommendation": "All good"}

model = genai.GenerativeModel(
    "gemini-2.0-flash",
    tools=[get_sensor_data, calculate_stress],
    system_instruction="You are SolarWine's farm monitoring assistant."
)

chat = model.start_chat()
response = chat.send_message(
    "Check block A3 temperature and tell me if the vines are stressed"
)
print(response.text)

LangChain vs LangGraph

Two popular frameworks for building agents and pipelines:

LangChain LangGraph
Concept Linear chain of steps Graph with cycles and branches
Best for Simple pipelines (RAG, QA) Complex agents, multi-step reasoning
Control Less (framework decides) More (you define the graph)
Example Document → Chunk → Embed → Answer Query → Search → Evaluate → (loop back or answer)
💡 Start Simple

For SolarWine, start with Gemini Function Calling — no framework needed. Define your sensor functions, let Gemini orchestrate them. Only reach for LangGraph when you need complex multi-step workflows with branching logic.

🛡️ Security Essentials

Before going to production, every LLM application needs these safeguards:

🔑

API Key Security

Never hardcode keys. Use environment variables or secret managers. Rotate keys regularly.

🛡️

Prompt Injection

Users can try to override your system prompt. Validate inputs, separate data from instructions.

⏱️

Rate Limiting

Protect your API budget. Set per-user and per-minute limits. Monitor usage dashboards.

🔒

Input Validation

Limit input length, sanitize special characters, block known attack patterns.

📊

Output Filtering

Check model output before showing to users. Filter PII, harmful content, and hallucinations.

🔐

Data Privacy

Know what data you send to cloud APIs. Use local models (Ollama) for sensitive SolarWine data.

Rate Limiter — Code Example

A simple in-memory rate limiter to protect your API budget. Drop this into any Flask/FastAPI app:

PYTHON 
from functools import wraps
import time
from collections import defaultdict

class RateLimiter:
    """Simple in-memory rate limiter."""

    def __init__(self, max_requests=10, window_seconds=60):
        self.max_requests = max_requests
        self.window = window_seconds
        self.requests = defaultdict(list)

    def is_allowed(self, user_id: str) -> bool:
        now = time.time()
        self.requests[user_id] = [
            t for t in self.requests[user_id]
            if now - t < self.window
        ]
        if len(self.requests[user_id]) >= self.max_requests:
            return False
        self.requests[user_id].append(now)
        return True

limiter = RateLimiter(max_requests=10, window_seconds=60)
